## 一、产品概述

### 1.1 产品定位与价值主张

#### 1.1.1 产品定位
**缅甸首个数字化房产交易与服务平台**，定位于通过技术赋能连接房产供需两端，重塑缅甸房产中介行业的信息透明度与服务标准。

#### 1.1.2 核心价值主张
| 维度 | 价值主张 | 用户痛点解决 |
|------|----------|--------------|
| **C端用户** | "真实房源，透明价格，全程可追溯" | 解决虚假房源泛滥、价格不透明、交易无保障问题 |
| **B端经纪人** | "协作共赢，按贡献分佣，职业化成长" | 解决恶性竞争、房源客源割裂、收入不稳定问题 |
| **平台方** | "构建缅甸房产数字化基础设施" | 建立行业标准，提升交易效率，沉淀数据资产 |

#### 1.1.3 差异化竞争策略
1. **验真体系差异化**: 针对缅甸市场"许可证地块"、"光头地契"等特殊产权形态建立专业验真流程
2. **协作网络本土化**: 引入ACN合作网络理念，适配缅甸经纪行业多品牌并存、门店分散的特点
3. **地图找房LBS化**: 结合缅甸城市特点（如仰光镇区制）优化地图找房体验

### 1.2 目标用户画像

#### 1.2.1 C端用户

**购房者（Buyer）**
```
【基础画像】
- 年龄: 28-45岁
- 职业: 企业主、外资企业员工、政府职员
- 收入水平: 月收入200-800万缅币（中高阶层）
- 分布: 仰光(60%)、曼德勒(20%)、内比都(10%)、其他(10%)

【行为特征】
- 决策周期长: 房产决策周期3-12个月
- 信息依赖强: 依赖经纪人获取房源和政策信息
- 信任阈值高: 对陌生经纪人信任度低，偏好熟人推荐
- 价格敏感: 汇率波动导致购房决策受宏观经济影响大

【核心需求】
- P0: 真实房源（虚假房源是最大痛点）
- P0: 价格透明（避免中介吃差价）
- P1: 交易安全（产权清晰、资金安全）
- P1: 专业服务（税费计算、贷款咨询）
- P2: 选房工具（地图找房、房贷计算器）
```

**租房者（Renter）**
```
【基础画像】
- 年龄: 22-35岁
- 职业: 企业职员、创业者、外籍人员
- 收入水平: 月收入80-300万缅币
- 租期: 短期(6-12个月)为主

【行为特征】
- 决策周期短: 1-2周完成决策
- 移动优先: 主要通过手机找房
- 即时沟通: 期望快速联系房东/经纪人

【核心需求】
- P0: 快速匹配（按区域、预算、户型筛选）
- P0: 真实图片（避免"照骗"）
- P1: 直接联系（减少中间环节）
- P2: 租房保障（押金保护、租约纠纷处理）
```

**房东（Owner）**
```
【基础画像】
- 类型: 个人业主、小型开发商、投资机构
- 持有量: 1-5套房产为主
- 分布: 以仰光为核心

【行为特征】
- 定价随意: 缺乏市场价格参考
- 渠道分散: 多平台发布房源
- 时间成本高: 带看协调困难

【核心需求】
- P0: 快速出租/出售
- P0: 价格指导（市场参考价）
- P1: 客源质量（筛选意向客户）
- P2: 交易代办（过户、税费处理）
```

#### 1.2.2 B端用户

**房产经纪人（Agent）**
```
【基础画像】
- 年龄: 25-40岁
- 从业经验: 0-5年（行业流动性高）
- 工作模式: 多平台兼职为主
- 收入水平: 月收入50-300万缅币（波动大）

【行为特征】
- 信息孤岛: 各经纪人房源信息不共享
- 恶性竞争: 抢单、切客现象普遍
- 服务非标准: 缺乏统一服务流程
- 工具落后: 依赖WhatsApp、Facebook获客

【核心需求】
- P0: 稳定客源（降低获客成本）
- P0: 协作分佣（打破零和博弈）
- P1: 房源管理（高效录入、更新）
- P1: 客户跟进（CRM工具）
- P2: 职业成长（培训、认证）
```

**团队长/店长（Team Leader）**
```
【基础画像】
- 管理规模: 5-20人团队
- 背景: 资深经纪人转型或自有门店
- 收入结构: 团队业绩提成+个人业绩

【核心需求】
- P0: 团队管理（人员、业绩、房源）
- P0: 分佣结算（透明、自动化）
- P1: 数据分析（团队效能）
- P1: 协作网络（跨店合作）
```

**地推人员（Ground Promoter）**
```
【基础画像】
- 工作性质: 兼职为主
- 核心任务: 房源开发、经纪人拉新

【核心需求】
- P0: 拉新激励（按结果结算）
- P0: 任务管理（地推目标、轨迹）
- P1: 即时到账（快速提现）
```

## 二、需求分析

### 2.1 市场背景分析

#### 2.1.1 缅甸房产市场特点

**宏观经济环境**
- **货币波动**: 缅币持续贬值，推动房产作为保值资产的投机需求
- **经济不确定性**: 2024年初市场降温，买家观望情绪浓厚
- **外资流入**: 缅甸资本在泰购房热潮不减，显示本地投资渠道有限

**市场结构特征**

**行业痛点总结**
| 痛点 | 严重程度 | 描述 |
|------|----------|------|
| 虚假房源泛滥 | ★★★★★ | 同一房源多平台重复发布，信息严重失真 |
| 产权核实困难 | ★★★★★ | 许可证地块产权追溯难，原始业主难寻 |
| 价格不透明 | ★★★★☆ | 中介吃差价现象普遍，无统一估价标准 |
| 经纪人职业化低 | ★★★★☆ | 流动性高，服务意识差，恶性竞争 |
| 交易无保障 | ★★★★☆ | 缺乏第三方资金监管，交易风险高 |
| 信息孤岛严重 | ★★★☆☆ | 各中介公司信息不互通，房源客源分散 |

#### 2.1.2 用户洞察

**缅甸用户特殊性**
1. **信任建立周期长**: 缅甸用户更倾向于通过熟人介绍建立信任，陌生人交易门槛高
2. **WhatsApp依赖**: 即时通讯以WhatsApp为主，需考虑IM集成
3. **移动支付起步**: 移动银行普及，但房产交易仍以现金为主
4. **多语言需求**: 部分外籍人士有英语界面需求

## 三、功能规格说明

### 3.1 C端功能模块

#### 3.1.1 账号体系

**功能背景**
缅甸移动互联网以手机为主，Facebook/WhatsApp普及率高。账号体系需支持手机号注册为主，同时考虑第三方登录降低注册门槛。

**用户故事**
- 作为购房者，我希望用手机号快速注册，以便开始使用平台找房
- 作为用户，我希望可以用Facebook登录，避免记忆新密码
- 作为平台，我需要验证用户身份真实性，建立信任基础

**功能描述**

| 功能点 | 描述 | 优先级 |
|--------|------|--------|
| 手机注册 | 缅甸手机号(+95)接收验证码注册 | P0 |
| 手机登录 | 验证码登录为主，密码登录为辅 | P0 |
| 实名认证 | 姓名+身份证号+身份证照片OCR | P0 |
| Facebook登录 | OAuth授权登录 | P1 |
| 找回密码 | 手机验证码重置密码 | P1 |

**业务规则**

```
【注册规则】
1. 手机号格式：必须+95开头，支持缅甸主要运营商(MPT, Telenor, Ooredoo, Mytel)
2. 验证码：6位数字，有效期5分钟，错误5次需等待30分钟
3. 同一手机号：24小时内最多发送5条验证码
4. 用户协议：注册需勾选同意用户协议和隐私政策

【实名认证规则】
1. 必传字段：真实姓名(缅语或英文)、身份证号、身份证正面照
2. OCR识别：自动提取身份证信息，人工复核
3. 审核时效：T+1工作日完成审核
4. 认证状态：未认证/审核中/已认证/认证失败
5. 限制：未认证用户可浏览，但不可预约带看

【登录规则】
1. 登录态有效期：30天
2. 多设备登录：允许，最多3台设备
3. 异常登录检测：新设备需短信验证
```

**输入输出**

```yaml
# 注册接口
Register:
  Input:
    phone: string  # +959123456789
    verifyCode: string  # 6位数字
    password: string  # 可选，默认验证码登录
  Output:
    userId: string
    token: string
    expiresAt: timestamp
    isNewUser: boolean

# 实名认证接口
VerifyIdentity:
  Input:
    realName: string
    idCardNumber: string
    idCardFront: file  # 身份证正面
    idCardBack: file   # 身份证反面(可选)
  Output:
    verifyId: string
    status: pending
    estimatedTime: timestamp  # 预计完成时间
```

**异常流程**
```
1. 手机号已注册: 提示"该手机号已注册，请直接登录"
2. 验证码错误: 提示"验证码错误，还剩X次机会"
3. 验证码过期: 提示"验证码已过期，请重新获取"
4. OCR识别失败: 转人工审核，预计24小时完成
5. 身份证已被使用: 提示"该身份证已绑定其他账号"
```

#### 3.1.2 首页门户

**功能背景**
首页是用户进入APP的第一触点，需要快速传达平台价值，并引导用户进入核心转化路径。

**用户故事**
- 作为用户，我希望打开APP就能看到感兴趣的房源推荐
- 作为新用户，我需要快速了解平台能帮我做什么
- 作为回访用户，我希望快速进入上次浏览的位置

**功能描述**

| 功能点 | 描述 | 优先级 |
|--------|------|--------|
| 房源推荐流 | 基于LBS的房源Feed流 | P0 |
| Banner轮播 | 运营位，支持跳转 | P0 |
| 快捷入口 | 买房/租房/发布/地图找房 | P0 |
| 搜索框 | 引导进入搜索页 | P0 |
| 城市切换 | 切换当前查看的城市 | P0 |

**业务规则**

```
【推荐算法规则(MVP简化版)】
1. 默认排序：综合排序 = 热度分(50%) + 新鲜度(30%) + 距离分(20%)
2. 热度分：近7天浏览量 + 收藏量*3 + 预约量*5
3. 新鲜度：房源上架时间，7天内为满分，超过30天降为0
4. 距离分：基于用户当前GPS位置计算直线距离
5. 个性化：MVP暂不做用户画像，二期引入协同过滤

【Banner规则】
1. 位置：首页顶部，轮播展示
2. 数量：最多5张
3. 尺寸：750*300px
4. 跳转：支持跳转到H5、房源详情、搜索结果页
5. 展示：支持按城市、用户类型定向展示

【快捷入口】
1. 买房：跳转到二手房搜索页，筛选条件为"出售"
2. 租房：跳转到租房搜索页，筛选条件为"出租"
3. 发布：跳转到房源发布页（未登录先登录）
4. 地图找房：跳转到地图找房页
```

**输入输出**

```yaml
# 首页推荐接口
HomeRecommend:
  Input:
    cityCode: string  # 城市编码
    districtCode: string  # 区域编码，可选
    page: int
    pageSize: int
  Output:
    banners:  # Banner列表
      - id: string
        image: string
        linkType: string  # house/detail/search/h5

**用户故事**

- 作为平台，我需要核实房源产权清晰

   - 特别提示交易风险
```

#### 3.2.3 ACN协作网络（重点详述）

**功能背景**
ACN（Agent Cooperate Network）是房产平台的核心差异化能力，通过协作分佣机制打破信息孤岛，促进经纪人合作共赢。

**用户故事**
- 作为房源录入人，我希望房源被成交后能获得分成
- 作为带看经纪人，我希望成交后能获得应有的佣金
- 作为平台，我需要公平合理的分佣机制激励协作

**功能描述**

**简化版ACN角色定义（MVP 5角色）**

```yaml
ACNRoles_MVP:
  # 房源方角色（2个）
  sourceSide:
    - code: ENTRANT
      name: 房源录入人
      desc: 首个将房源录入平台的经纪人
      commissionRatio: 15%
      
    - code: MAINTAINER
      name: 房源维护人
      desc: 负责房源日常维护、陪同看房的经纪人
      commissionRatio: 15%
      
  # 客源方角色（3个）
  clientSide:
    - code: INTRODUCER
      name: 客源转介绍
      desc: 首次将客户推荐给平台的经纪人
      commissionRatio: 10%
      
    - code: ACCOMPANIER
      name: 带看人
      desc: 实际陪同客户看房的经纪人
      commissionRatio: 10%
      
    - code: CLOSER
      name: 成交人
      desc: 最终促成签约成交的经纪人
      commissionRatio: 40%
  
  # 平台服务费
  platformFee: 10%  # 平台抽取服务费
  
  # 特殊情况
  special:
    singleAgent:  # 单一经纪人完成全部流程
      commissionRatio: 90%  # 平台10%，经纪人90%
```

**ACN协作流程**

```
┌─────────────────────────────────────────────────────────────────┐
│                     ACN协作流程                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   Step 1: 房源录入                                               │
│   ┌─────────────────────────────────────────────────────────┐  │

│   │ 经纪人B将客户C推荐给该房源 → 成为【客源转介绍】           │  │
│   └─────────────────────────────────────────────────────────┘  │
│                              │                                  │
│                              ▼                                  │
│   Step 3: 带看服务                                               │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │ 经纪人D陪同客户C看房 → 成为【带看人】                     │  │

│                              │                                  │
│                              ▼                                  │
│   Step 4: 成交签约                                               │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │ 经纪人E促成客户C签约 → 成为【成交人】                     │  │

│                              │                                  │
│                              ▼                                  │
│   Step 5: 分佣结算                                               │

│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

**业务规则**

```
【角色确认规则】
1. 房源录入人：系统自动确认，以首次录入为准
2. 房源维护人：录入人默认，可转让给其他经纪人（需双方确认）
3. 客源转介绍：客户首次在平台登记意向时确定，保护期30天
4. 带看人：实际带看时签到确认
5. 成交人：成交申报时确认，需其他方确认无异议

【保护期规则】
1. 房源保护：录入后30天内仅录入人可成交，30天后开放给所有人
2. 客源保护：客源登记后30天内归转介绍人，过期后释放为公客
3. 带看保护：带看后30天内成交，带看人获得分佣

【成交申报规则】
1. 成交人需在签约后24小时内发起成交申报
2. 需上传合同照片、填写成交价格、佣金金额
3. 需选择/添加参与方（录入人、维护人、转介绍人、带看人）
4. 参与方需在48小时内确认，超时自动确认
5. 有异议可发起申诉，平台介入仲裁

【分佣结算规则】
1. 结算时点：交易完成（过户/交房）后T+7天
2. 结算方式：平台自动计算，各方确认后入账
3. 提现规则：T+1到账，最低提现金额1万缅币
4. 税费处理：平台代扣代缴个人所得税（如法规要求）

【违规处罚】
1. 跳单：发现后封号处理，扣罚全部佣金
2. 虚假申报：扣除当月全部佣金，警告/封号
3. 恶意抢占角色：警告，情节严重封号
```

• 有房源：日新增/维护房源的经纪人
• 有客户：日跟进客户的经纪人
• 有带看：日完成带看的经纪人
• 有成交：月有成交的经纪人

效能指标：
• 人均房源量：总房源数/活跃经纪人
• 人均客户量：总客户数/活跃经纪人
• 人均带看量：总带看数/活跃经纪人
• 人均GMV：总GMV/活跃经纪人
```

### 4.4 埋点实施建议

```
【技术方案】
1. 埋点SDK：使用开源方案（如神策、GrowingIO）或自建
2. 上报方式：批量上报，每30秒或退出时上报
3. 本地缓存：失败时本地缓存，下次启动补报
4. 采样率：全量上报（数据量可控情况下）

【关键配置】
- 用户标识：user_id（登录后）/ device_id（匿名）
- 会话标识：session_id，30分钟无操作视为新会话
- 时间戳：客户端时间戳 + 服务端时间戳校准
- 设备信息：设备型号、系统版本、APP版本
```

# 第七章：安全规范

> **v2.0 新增章节** — 基于代码审查发现的安全隐患，制定平台安全基线。

## 7.1 JWT 安全规范

### 7.1.1 密钥管理
- JWT 签名密钥**必须**通过环境变量注入，禁止硬编码于代码或配置文件
- 生产密钥长度 ≥ 256 位，使用 `openssl rand -hex 32` 生成
- 密钥轮换周期：每 90 天或发生安全事件时立即轮换
- 开发/测试/生产环境使用不同密钥集

```yaml
# config.yaml 示例（值从环境变量读取）
jwt:
  access_secret: ${JWT_ACCESS_SECRET}
  refresh_secret: ${JWT_REFRESH_SECRET}
  access_expire: 2h
  refresh_expire: 30d
```

### 7.1.2 Token 生命周期

| Token 类型 | 有效期 | 存储位置 |
|-----------|--------|---------|
| Access Token | 2 小时 | 内存（不持久化） |
| Refresh Token | 30 天 | flutter_secure_storage（加密） |

### 7.1.3 Token 黑名单（登出机制）
- 登出时将 Access Token 写入 Redis 黑名单，TTL = Token 剩余有效期
- 所有需要认证的接口在 JWT 验证前先查 Redis 黑名单
- Redis Key 格式：`blacklist:{token}` — 已实现（2026-03-20 P0修复）

## 7.2 API 安全规范

### 7.2.1 认证中间件要求
- 所有 `/v1/*` 路由中，需认证接口**必须**经过 `AuthMiddleware`
- 中间件从 JWT Claims 读取 `user_id`，**禁止**从请求参数或 Header 中直接信任 user_id
- 中间件须验证 token type = "access"（拒绝 refresh token 直接访问 API）
- **已修复（2026-03-20）**：原中间件硬编码 `user_id=1` 的 P0 安全漏洞

### 7.2.2 验证码防刷
- 同一手机号发送验证码：60 秒冷却，24 小时内最多 10 次
- 同一 IP 发送验证码：24 小时内最多 50 次
- 实现：Redis 计数器 + TTL，Key 格式 `sms:phone:{phone}` / `sms:ip:{ip}`

### 7.2.3 SQL 注入防护
- 禁止使用 `db.Raw()` 拼接用户输入字符串
- 地图聚合查询等复杂场景使用参数化查询（`?` 占位符）
- 定期使用 sqlmap 或类似工具扫描 API

### 7.2.4 文件上传安全
- 图片上传：验证 MIME Type（白名单：jpg/png/webp），限制大小 ≤ 10MB
- 上传文件存储于独立 MinIO bucket，不与应用服务器同域
- 禁止上传可执行文件；文件名使用 UUID 重命名，不保留原始文件名

## 7.3 数据隐私规范

### 7.3.1 敏感字段处理

| 字段 | 存储方式 | 传输方式 |
|------|---------|---------|
| 密码 | bcrypt hash（cost=12） | 不传输（仅接收） |
| 手机号 | 明文（业务需要） | API 响应中脱敏显示（138****8888） |
| 身份证号 | 明文（实名认证） | 不在列表接口返回 |
| Token | Redis 存明文 | HTTPS 传输 |

### 7.3.2 数据访问控制
- 经纪人只能查看/修改自己的房源（`agent_id = current_user_id`）
- 验真员只能操作分配给自己的任务
- Admin API 须单独认证，不共用 C/B 端 JWT

## 7.4 传输安全
- 生产环境强制 HTTPS（TLS 1.2+），HTTP 请求 301 重定向
- HSTS Header：`Strict-Transport-Security: max-age=31536000; includeSubDomains`
- API 响应设置 `X-Content-Type-Options: nosniff`
- Web Admin 设置 `Content-Security-Policy` 防 XSS

# 第九章：技术选型决策

> **v2.0 新增章节** — 记录 PRD 评审阶段确认的关键技术选型，作为开发实施依据。

## 9.1 即时通讯（IM）SDK

**选型结果：环信（Easemob）**

| 对比项 | 环信 | 融云 |
|-------|------|------|
| 缅语支持 | Unicode 完整支持 | Unicode 完整支持 |
| 中文文档 | 完善 | 完善 |
| Flutter SDK | 官方维护 | 官方维护 |
| 免费额度 | 100 MAU 免费 | 50 MAU 免费 |
| 选择原因 | 文档更完善，社区活跃 | — |

**集成要点：**
- 用户注册时同步在环信创建 IM 账号（`POST /users` 环信 REST API）
- 后端 `GET /v1/im/token` 返回环信 Token，Flutter 端用于 SDK 登录
- 消息类型：文本、图片、房源卡片（自定义消息）
- 会话列表、未读数由环信 SDK 本地维护

## 9.2 地图 SDK

**选型结果：flutter_map + OpenStreetMap（OSM）**

| 对比项 | flutter_map + OSM | Google Maps |
|-------|------------------|-------------|
| 费用 | 完全免费 | 按量计费（$7/1000次） |
| 离线支持 | 可下载缅甸地图瓦片 | 需付费 |
| 缅甸地图精度 | 良好（OSM 社区维护） | 优秀 |
| 合规风险 | 低 | 中（Google 数据出境） |

**集成规范：**
```yaml
# pubspec.yaml
flutter_map: ^6.0.0
latlong2: ^0.9.0
```
- 地图瓦片：`https://tile.openstreetmap.org/{z}/{x}/{y}.png`
- 仰光初始中心坐标：LatLng(16.8661, 96.1951)，Zoom = 12
- 地图标注：三级聚合（城市级 → 区级 → 单点），按价格区间着色

## 9.3 云存储

**选型结果：MinIO 自建**

| 对比项 | MinIO 自建 | AWS S3 |
|-------|-----------|--------|
| 数据合规 | 数据留在缅甸境内 | 数据出境风险 |
| 成本 | 仅服务器成本 | 按用量计费 |
| S3 API 兼容 | 完全兼容 | 原生 |
| 运维复杂度 | 中（需自维护） | 低 |

**部署规范：**
- MinIO 独立部署，与应用服务器隔离
- Bucket 命名：`myanmar-property-images`、`myanmar-property-docs`
- 图片访问通过 CDN 代理（可选：Cloudflare 免费计划）
- 上传使用后端预签名 URL（Presigned URL），避免前端直接持有 MinIO 密钥

## 9.4 推送通知

**选型结果：Firebase Cloud Messaging（FCM）**

- Android 使用 FCM，iOS 使用 APNs（通过 FCM 统一接入）
- 触发场景：预约确认/取消、IM 新消息（离线时）、房源状态变更、ACN 分佣确认请求

# 第十章：地推员完整规格

> **v2.0 新增章节** — 地推是平台冷启动关键，本章定义完整的地推员业务规格。

## 10.1 地推员角色定义

**地推员（Promoter）**：受平台雇用或合作，在线下推广平台、发展新用户（C端买家/B端经纪人）的角色。

| 属性 | 说明 |
|------|------|
| 来源 | 平台招募，通过后台审核开通 |
| 激励方式 | 按有效新用户数 + 成交单数计算佣金 |
| 管理后台 | Web Admin → 地推管理模块 |

## 10.2 地推员注册与审核流程

```
地推员申请 → 填写身份信息 + 区域 + 联系方式
     ↓
平台审核（1–2 工作日）
     ↓
审核通过 → 生成专属邀请码 + 二维码
     ↓
地推员开始推广
```

**申请字段：**
- 姓名（Myanmar / English）
- 手机号（缅甸格式）
- 身份证照片（正反面）
- 推广区域（Township 级别，可多选）
- 推荐人（其他地推员邀请码，选填）

## 10.3 拉新任务规则

### 10.3.1 有效新用户定义
- 通过地推员专属二维码/邀请码注册
- 注册后 7 天内完成首次搜房（C端）或发布首个房源（B端）

### 10.3.2 佣金标准（可在后台配置）

| 类型 | 佣金 |
|------|------|
| 有效 C 端新用户 | 500 缅元 |
| 有效 B 端新经纪人 | 2,000 缅元 |
| 经纪人通过 ACN 首次成交 | 额外 5,000 缅元奖励 |

### 10.3.3 任务进度查看（B端 App 地推员视图）
- 今日拉新数 / 本月拉新数
- 待结算佣金 / 已结算佣金
- 推广区域排行榜

## 10.4 佣金结算流程

```
月末统计有效新用户 → 平台审核（5 工作日）→ 佣金打款（KBZ Pay / 银行转账）
```

- 结算周期：每月 1 日结算上月佣金
- 最低提现门槛：10,000 缅元
- 提现方式：KBZ Pay（一期）、银行转账（二期）

## 10.5 防刷单机制

- 同一手机号/设备 ID 只能被一个邀请码绑定
- 注册后账号活跃度验证（7 天内有实质操作）
- 异常注册自动标记（同 IP 批量注册、模拟器设备）
- 地推员连续 3 个月零有效用户，自动冻结资格

# 第十一章：多语言规范

> **v2.0 新增章节** — 缅甸市场核心语言为缅语，本章定义 i18n 实现规范。

## 11.1 语言策略

| 优先级 | 语言 | 覆盖范围 |
|--------|------|---------|
| P1 | 缅语（my） | 全部 UI，C端必须 |
| P2 | 英语（en） | 全部 UI，通用 fallback |
| P3 | 中文（zh） | 可选，服务中国投资者 |

**默认语言选择逻辑：**
1. 用户手动设置语言（存 SharedPreferences）
2. 读取设备系统语言
3. 以上均未命中，使用缅语（my）

## 11.2 Flutter i18n 实现规范

### 11.2.1 使用 Flutter ARB 文件

```
lib/
└── l10n/
    ├── app_en.arb    # 英语
    ├── app_my.arb    # 缅语
    └── app_zh.arb    # 中文（可选）
```

### 11.2.2 ARB 文件示例

```json
{
  "@@locale": "my",
  "appName": "မြန်မာ အိမ်ခြံမြေ",
  "searchHint": "ရန်ကုန် ၊ မန္တလေး...",
  "loginTitle": "ဝင်ရောက်ပါ",
  "phoneNumber": "ဖုန်းနံပါတ်",
  "sendCode": "ကုဒ် ပို့ပါ"
}
```

### 11.2.3 pubspec.yaml 配置

```yaml
flutter:
  generate: true

dependencies:
  flutter_localizations:
    sdk: flutter
  intl: ^0.18.0
```

### 11.2.4 代码规范
- **禁止**在 Widget 中硬编码字符串，全部使用 `AppLocalizations.of(context).xxx`
- **例外**：纯英文专业术语（如 "GPS"、"ACN"、"MinIO"）可硬编码
- 字符串 Key 使用驼峰命名，按页面模块分组（如 `homeSearchHint`、`detailContactAgent`）

## 11.3 缅语字体支持

- 使用 **Pyidaungsu** 字体（开源，缅语官方推荐字体）
- Flutter 字体配置：

```yaml
flutter:
  fonts:
    - family: Pyidaungsu
      fonts:
        - asset: assets/fonts/Pyidaungsu-2.5.3_Regular.ttf
        - asset: assets/fonts/Pyidaungsu-2.5.3_Bold.ttf
          weight: 700
```

## 11.4 后端多语言规范

- 错误消息：后端返回错误码，前端根据码查本地 ARB 文件显示对应语言消息
- 数据字段：房源标题、描述等支持多语言字段（`title_my`、`title_en`）
- API 接受 `Accept-Language` Header，排序/筛选结果优先展示对应语言内容

## 11.5 上线语言里程碑

| 时间 | 目标 |
|------|------|
| MVP 上线 | 缅语 + 英语双语完整覆盖 |
| 上线后 3 个月 | 中文支持（服务中国投资者） |
| 上线后 6 个月 | 语言切换记账到用户账号（多设备同步） |

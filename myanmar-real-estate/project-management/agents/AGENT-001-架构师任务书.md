### 3.2 API规范要求

```yaml
# RESTful API 设计规范

# 1. URL规范
#    - 小写字母，连字符分隔
#    - 资源名词复数形式
#    - 版本号: /v1/

# 2. HTTP方法
#    GET    - 查询
#    POST   - 创建
#    PUT    - 全量更新
#    PATCH  - 部分更新
#    DELETE - 删除

# 3. 响应格式
{
  "code": 0,           // 0成功，非0错误码
  "message": "success", // 提示信息
  "data": {},          // 业务数据
  "requestId": "xxx"   // 请求ID，用于追踪
}

# 4. 分页格式
{
  "code": 0,
  "data": {
    "list": [],
    "total": 100,
    "page": 1,
    "pageSize": 20
  }
}

# 5. 错误码规范
#    1000-1999: 系统级错误
#    2000-2999: 用户相关错误
#    3000-3999: 房源相关错误
#    4000-4999: 交易相关错误
#    5000-5999: 预约相关错误
```

## 四、技术选型要求

### 4.1 必做技术选型

| 领域 | 选项 | 决策要求 |
|------|------|----------|
| 后端框架 | Go + Gin | 确定版本，制定规范 |
| 数据库 | PostgreSQL | 版本选择，主从方案 |
| 缓存 | Redis | Cluster方案 |
| 消息队列 | Redis/RabbitMQ | 根据需求选择 |
| 搜索 | PostgreSQL全文搜索/ES | 初期可用PG，规划ES扩展 |
| IM | 环信/融云/自建 | 评估后选择 |
| 地图 | Google Maps/Mapbox | 考虑缅甸可用性 |
| 存储 | MinIO/AWS S3 | 成本评估 |
| 监控 | Prometheus + Grafana | 部署方案 |

### 4.2 地图聚合算法要求

```yaml
# 地图聚合算法设计要点

Level 1 (Zoom 8-11): 镇区级聚合
  - 聚合粒度: District
  - 显示: 镇区名 + 均价 + 房源数
  - 计算: 按district_code分组

Level 2 (Zoom 12-14): 商圈级聚合
  - 聚合粒度: Community
  - 显示: 商圈名 + 均价 + 房源数
  - 计算: 按community_id分组

Level 3 (Zoom 15+): 单个房源
  - 显示: 价格标签
  - 数据: 单个房源详情

技术实现:
  - 使用GeoHash进行空间索引
  - Redis缓存聚合结果(5分钟)
  - 可视区域+边界缓冲查询
```
